在实际的 Go 语言开发中：**要创建一个真正能正常使用的 channel，只能用 `make`。**

这背后涉及到 Go 语言中一个非常经典的面试考点，我们可以从以下几个维度来彻底理解它：

### 1. 如果不用 `make`，只用 `var` 声明会怎样？

如果你只声明通道，而不使用 `make`：

```go
var ch chan int

```

此时 `ch` 是一个引用类型，它没有被初始化，其默认的零值是 **`nil`**。
在 Go 语言中，操作一个 `nil` 通道是非常危险的：

* **往 `nil` 通道发数据：** 协程会永远阻塞（死锁）。
* **从 `nil` 通道读数据：** 协程也会永远阻塞（死锁）。
* **关闭 `nil` 通道：** 直接导致程序崩溃（Panic）。

### 2. 面试高频追问：能用 `new` 来创建 channel 吗？

面试官经常会顺势考你 `make` 和 `new` 的区别。
如果你尝试用 `new` 来创建：

```go
ch := new(chan int)

```

语法上不会报错，但是**完全没用**。

* `new` 的作用仅仅是**分配内存**，但不做初始化，并且返回的是指向这块内存（零值）的**指针**。
* 对于 channel 来说，`new(chan int)` 只是分配了一个指针 `*chan int`，这个指针指向的值依然是未初始化的 `nil`。由于底层负责控制并发的结构体（`hchan`，包含锁和环形队列等）根本没有被真正构建出来，所以它完全无法用来进行通信。

### 3. 为什么必须是 `make`？

Go 语言官方明确规定，**`make` 是专门用来为 `slice`（切片）、`map`（字典） 和 `channel`（通道）这三种特定类型分配内存并进行“初始化”的内置函数**。

当你执行 `make(chan int)` 时，Go 语言底层真正做的事情是：

1. 在堆内存中为你分配一块空间。
2. 构造出一个叫做 `hchan` 的底层结构体对象。
3. 初始化这个结构体里面的互斥锁（mutex）、等待队列（sendq / recvq）以及环形缓冲区（如果有容量的话）。
4. **返回这个结构体本身（而不是指针）**，让你可以在代码里直接进行 `<-` 收发操作。

**总结：** 只要你想用 channel、map 或者 slice，并且打算往里面放数据，**无脑用 `make` 就对了**。
